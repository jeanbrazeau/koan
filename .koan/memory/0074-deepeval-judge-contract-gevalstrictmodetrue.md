This entry documents the judge-LLM contract for the DeepEval-based koan eval harness in `evals/scorers.py`. Four dated revisions shape the current state.

On 2026-04-22, Leon chose `GEval(name=..., criteria=<rubric-body>, strict_mode=True, model=JUDGE_MODEL)` for per-phase rubrics, with `JUDGE_MODEL = GeminiModel(model="gemini-3-pro")` as a shared module constant. Rationale: binary PASS/FAIL matched the rubric files' existing "PASS or FAIL on the last line" tail contract, and passing the full rubric body via `criteria=` preserved the migration's "don't change rubric content" constraint. `DAGMetric` for per-criterion decomposition was deferred. `LiteLLMModel('google/gemini-3.1-pro-preview')` was rejected (plan-review caught the nonexistent `3.1` family).

On 2026-04-23 morning, during an intake-phase redesign, Leon adopted DAGMetric for a new `RubricCompliance` dimensional metric -- each rubric bullet becoming a `BinaryJudgementNode` aggregated via VerdictNodes.

On 2026-04-23 afternoon, plan-review reversed the DAGMetric choice based on a reading of the root-node constraint. Observation at that time: `DeepAcyclicGraph.__init__` in deepeval 3.9.7 raises `ValueError("You cannot provide more than one root node when using 'BinaryJudgementNode' or 'NonBinaryJudgementNode' in root_nodes.")` whenever more than one binary judgement appears as root. The replacement was a custom `RubricComplianceMetric(BaseMetric)` that takes no criteria at construction, reads `rubric_criteria` from `LLMTestCase.additional_metadata` inside `a_measure()`, and issued one `JUDGE_MODEL.a_generate_with_schema(prompt, CriterionVerdict)` call per criterion via `asyncio.gather`. Verdicts were averaged into a pass-rate; `threshold=1.0` preserved all-or-nothing binary semantics while keeping the numeric score visible in `score_breakdown` for partial-failure diagnosis. `GEval` for CrossPhaseCoherence was retired in the same afternoon revision and replaced with `CrossPhaseCoherenceMetric(BaseMetric)` reading `rubric_body` from `additional_metadata`. The conversion was needed because `evaluate()` applies the same metrics list to every test case, forcing metrics to be construction-parameterless singletons rather than per-row factory instances. `strict_mode=True` binary semantics preserved across both metrics. `JUDGE_MODEL` upgraded to `GeminiModel(model="gemini-3-pro-preview")` in the same revision.

On 2026-04-23 late, two findings reopened the DAGMetric direction. First, the Confident AI dashboard renders "No verbose logs available" for every custom BaseMetric row because its frontend carries a CSS class `disableVerbose` that hides verbose content for non-DeepEval-authored metrics (Leon verified via payload intercept -- verbose strings are present in the API request body). DAGMetric's `construct_node_verbose_log()` is the only path to per-criterion dashboard visibility. Second, the `asyncio.gather` pattern across criteria combined with `AsyncConfig(max_concurrent=100)` at evaluate() level produced a thundering herd of 136+ concurrent Gemini API calls; that pattern was retired in favor of sequential per-criterion iteration inside `a_measure()` and `max_concurrent=10` at evaluate() level. A third correctness fix folded in: `GeminiModel.a_generate_with_schema()` returns a `(result, cost)` tuple, not a bare result, so every call site in `evals/scorers.py` now unwraps with `verdict, _cost = await JUDGE_MODEL.a_generate_with_schema(...)`. Leon's action on 2026-04-23 late: refactor `RubricComplianceMetric` to a DAGMetric with one `BinaryJudgementNode` per rubric criterion rooted under a single `TaskNode` (to satisfy the single-root-node constraint that triggered the earlier reversal), preserving threshold=1.0 all-or-nothing semantics via the DAG's verdict aggregation. The custom BaseMetric variant is retained only for metrics whose dashboard verbose output is not part of the observability story.
