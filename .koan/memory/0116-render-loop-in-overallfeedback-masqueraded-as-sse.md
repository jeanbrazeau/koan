On 2026-04-27 the memory-curation overlay (`frontend/src/components/organisms/CurationTakeover.tsx`) failed to dismiss after the user clicked Submit batch. The backend received the submission (subsequent submits returned 409 Conflict), the orchestrator advanced into thinking, and a browser refresh removed the overlay -- so the projection was being cleared correctly server-side. The browser console showed the misleading log line `'Patch failed, reconnecting for fresh snapshot:'` from `frontend/src/sse/connect.ts:45` together with React's `'Maximum update depth exceeded'`. Investigation identified the root cause: the catch in `connect.ts` wraps the entire `applyPatch` + Zustand `setState` chain, so any React throw raised during the resulting render commits is rethrown as if `applyPatch` itself had failed. The actual throw was a render-loop abort whose stack pointed at `OverallFeedback.tsx:36-38`, where `useEffect(() => onFileIdsChange?.(attach.fileIds), [attach.fileIds, onFileIdsChange])` had two unstable deps: (a) `attach.fileIds` was recomputed as a fresh array on every render at `useFileAttachment.ts:122` (`files.filter(...).map(...)` with no memoization), and (b) `onFileIdsChange` was an inline arrow at `CurationTakeover.tsx:130` (`onProposalFileIdsChange={(id, ids) => setDecisionFileIds(prev => ({ ...prev, [id]: ids }))}`) re-created every render. Either dep alone re-fired the effect on every render; the effect called the parent setter, which spread into `decisionFileIds` and committed new state, which re-rendered the parent with a fresh inline arrow, looping until React aborted. Because the loop fired on every projection patch while the overlay was open (streaming `toolInputDelta` patches were the trigger observed in logs, not the `memory_curation_cleared` patch), the EventSource was closed by the catch (silently -- `es.close()` does not fire `onerror`, so the App.tsx reconnect path never ran) and the store's `run.activeCurationBatch` stayed at its stale non-null value until manual refresh fetched a fresh snapshot. On 2026-04-27 the fix was applied: `useFileAttachment.fileIds` now uses `useMemo(..., [files])` so identity is stable across renders that don't change file state, and `OverallFeedback.tsx` drops `onFileIdsChange` from its effect deps with an `eslint-disable-next-line react-hooks/exhaustive-deps` -- the effect closure re-binds to the latest callback on every render, so re-firing on callback identity changes is semantic noise (the parent already had the same `fileIds` value when its prior closure was current). The lesson: a thrown error inside `connect.ts`'s patch handler should be treated as suspect for being a downstream React render error, not a JSON-Patch resolution failure -- the misleading `'Patch failed'` log line will need refining if this happens again.
