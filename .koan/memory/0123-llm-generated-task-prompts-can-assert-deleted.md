This entry records a misread caught during intake of the koan initiative+discovery workflow-presets initiative on 2026-04-29. The user-supplied task prompt (generated by Claude Desktop, per the prompt's preface) asserted that "the koan_artifact_propose tool is already orchestrator-allowed in every phase per koan/lib/permissions.py, and the frontend already renders the artifact-review surface that backs it." Both claims were false: the koan_artifact_propose backend handler had been deleted on 2026-04-26 (along with the artifact-review frontend molecules ArtifactReviewPin, ReviewBlock, ReviewComment, ReviewCommentInput, ReviewEvent the same day). Compounding the misread, AGENTS.md's MCP-permission-fence table still listed koan_artifact_propose as a live tool -- the doc had drifted from the live fence in koan/lib/permissions.py. Intake caught the contradiction by direct grep for "koan_artifact_propose" against koan/web/mcp_endpoint.py (zero handler matches) and koan/lib/permissions.py (only deletion-context comments) before trusting the task prompt. The discrepancy reshaped the design: the load-bearing "Approved gate via koan_artifact_propose" mechanism the task prompt described could not be implemented as written. Root cause: AGENTS.md's permission-fence table is hand-maintained and lagged the deletion; LLM-generated task descriptions consume AGENTS.md as authoritative. Lesson for future intake runs: when a task description (especially LLM-generated) cites tool availability or UI infrastructure, verify against the source-of-truth files (koan/lib/permissions.py for the fence; koan/web/mcp_endpoint.py for tool handlers; frontend/src/ for UI components) and recent git log for deletion commits, NOT against AGENTS.md tables alone.
