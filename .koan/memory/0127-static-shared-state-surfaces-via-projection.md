---
title: Static shared state surfaces via projection Settings.*, not via new HTTP endpoints
type: decision
created: '2026-04-29T07:23:53Z'
modified: '2026-04-29T07:23:53Z'
---

The koan projection at `koan/projections.py` (the `Settings`, `Run`, and SSE-broadcast snapshot/patch wire format) holds shared state visible to both the koan backend and the frontend Zustand store. On 2026-04-29, during a plan-workflow run that introduced backend-driven workflow discovery for the NewRunForm landing page, Leon directed that the workflows registry from `koan/lib/workflows.py:WORKFLOWS` be exposed as `Settings.workflows: list[WorkflowInfo]` populated by a new `workflows_listed` initial event pushed from `_push_initial_config_events` in `koan/web/app.py`, rather than via a new HTTP endpoint such as `/api/workflows`. The agent had initially proposed a dedicated GET endpoint, offering naming variants (`/api/workflows`, `/api/config/workflows`, `/api/discover`); Leon redirected with the reasoning that the projection is the canonical shared-state channel between UI and backend, and that a separate HTTP surface fits less well within the existing architecture even when the data is static. Two alternatives were rejected: (1) a dedicated HTTP endpoint -- duplicates the channel and creates a parallel discovery surface that bypasses the SSE-driven projection; (2) populating `Settings.workflows` via `Field(default_factory=...)` at construction time -- works for static data but diverges from the established `_push_initial_config_events` pattern, where installations, profile registry, default profile, and scout concurrency all flow through initial events with their own fold cases. The decision implies a procedural rule: when adding static configuration data that the frontend will read on the landing page, route it through `Settings.*` and an initial event in `_push_initial_config_events`, not through a new HTTP endpoint or a default factory. The same change deleted `Run.available_workflows` (previously rebuilt by the `workflow_selected` fold) so that `Settings.workflows` is the single source of truth for the workflows registry across both the run-time `/workflow:<name>` palette and the landing-page form.
